# MoanSwipe AI Planning Guidelines v1.0

## 0. 目的

本ガイドラインは、MoanSwipe における AI 主導開発の計画書作成・レビュー・実装進行の基準を定義する。

目的は以下の 4 点である。

- 現行要件と実装の整合性を維持する
- menubar / 入力監視 / 音声再生という不安定要素を、段階的な検証で制御する
- 短絡的な実装や過剰設計を防ぐ
- フェーズごとに小さく検証し、失敗時の切り戻しを容易にする

## 4. Do / Don't

### 4.1 Do

- 実装前に `PLAN.md` と Xcode プロジェクト設定を確認する
- menubar アプリとしての完成経路を最短にする
- 1 フェーズ 1 目的で小さく進める
- 既存ファイルに責務を寄せ、不要な新規ファイルを増やさない
- API / OS 依存で不確実な要素はスパイクとして切り分ける
- 失敗パス、権限不足、音源欠損を Gate 条件に含める

### 4.2 Don't

- `Force Click` や `Pinch` を検証前に MVP 必須として固定しない
- menubar 化前に見た目の UI 作り込みへ進まない
- 入力検出ロジックと音声再生ロジックを直接結合しない
- 一時ログ、デバッグフラグ、暫定分岐を計画に残したまま放置しない
- 1 フェーズで複数の目的を同時達成しようとしない

---

## 4.5 緊急時例外適用ルール

### 例外適用条件

- menubar 起動不能など、アプリの基本機能が停止している
- Xcode 設定や sandbox 制約により、通常手順では検証不能
- 要件文書と実装前提に解決不能な矛盾がある
- ユーザー影響が大きく、通常フェーズ進行では復旧が遅すぎる

### 例外適用時の必須要件

- 例外適用理由を明文化する
- どのフェーズを一時中断したか記録する
- 正常系へ戻す段階的復帰計画を作る
- ロールバックまたは一時無効化の方針を明記する

## 6. AI 主導開発で頻出する落とし穴

### 6.1 よくある失敗

- menubar アプリなのに `WindowGroup` 前提の設計を引きずる
- イベント取得 API の制約確認前に上位アーキテクチャを固める
- 入力種別ごとに分岐を増やしすぎて Audio 層が肥大化する
- 成功パスだけ確認して、権限不足や音源欠損を見落とす
- 連続スクロールで音声が過剰発火し、UX が破綻する
- 仮ログや検証コードを残したまま次フェーズへ進む

### 6.2 リスク低減プロンプト例

- 「`PLAN.md` と現行実装に対する整合性差分を先に列挙し、差分ごとに実装前提か技術スパイクかを分類してください。」
- 「入力取得、生イベント正規化、音声カテゴリ選択、再生を層として分離し、状態更新の発火源を一本化してください。」
- 「暫定ログ・検証用コード・TODO は、各フェーズの Gate 条件に削除または課題化を必須項目として含めてください。」
- 「成功パスと失敗パスの動作確認リストを作り、OFF 時無反応、音源欠損、権限不足を必ず確認項目に入れてください。」
- 「パフォーマンスや UX 副作用として、入力遅延、過剰発火、同一音連打、メニュー状態不整合を事前に列挙してください。」

## 7. MoanSwipe 固有の短絡防止テンプレ

### 7.1 menubar 化

- `WindowGroup` に依存したまま進めない
- Dock 主体の UI に逆戻りしない
- 最初の Gate は「メニューバーから ON/OFF と Quit が操作できること」とする

### 7.2 入力監視

- click / tap / scroll を一気に取ろうとしない
- 非アクティブ時監視が必要なら、先に実用上の制約を確認する
- `Force Click` は検証完了まで特殊機能扱いに留める

### 7.3 音声再生

- 最初から Voice Pack 管理を複雑化しない
- まず単発再生を安定させ、その後プール化する
- 直前音回避とクールダウンは Audio 層に集約する

### 7.4 UX 副作用チェック

- OFF 切替直後に音が鳴らないか
- 連続スクロールで発火過多にならないか
- 同じ音ばかり繰り返さないか
- menubar メニュー状態と内部設定がズレないか

## 8. 計画書の必須構成

すべてのフェーズ計画書は、以下の構成を必須とする。

1. **SSOT参照宣言**
2. **既存実装の類似参照**
3. **問題一覧（Issue List）**
4. **修正フェーズ（Phase）**
5. **Gate 条件（Exit Criteria）**
6. **回帰 / 副作用チェック**
7. **変更履歴**

---

## 9. Issue ID 体系

MoanSwipe では以下の ID 体系を使う。

- `C-` Configuration / project setup
- `M-` Menubar / application shell
- `I-` Input / gesture detection
- `A-` Audio / playback pipeline
- `S-` Settings / persistence / menu state
- `D-` Documentation / planning / review
- `K-` Known risk / spike / investigation

非標準 ID は使用しない。

---

## 10. MECE 検査

計画書完成時に、以下を必ず実施し結果を記録する。

### 検査A: Issue → Phase 対応

- すべての Issue に対応する Phase が存在する
- すべての Phase に対応する Issue が存在する

### 検査B: 既存実装との差分

- 参照実装との差分を列挙する
- 差分がある場合は理由と回帰リスクを記述する

### 検査C: DRY / KISS

- DRY: 重複処理を 1 か所へ寄せる設計になっているか
- KISS: 既存テンプレより複雑になる理由が明文化されているか

---

## 11. フェーズ管理と閾値

AI 実装は小さく段階的に進める。

### 11.1 フェーズ閾値

- 1 フェーズで触るファイル数: 最大 3
- 新規ファイル作成: 最大 1
- 1 フェーズの主目的: 1 つのみ

### 11.2 逸脱時ルール

閾値を超える場合は、実装を進める前に計画を分割修正する。計画修正なしに複数目的をまとめて実装しない。

---

## 12. 計画違反の処遇

### 12.1 違反の定義

- `PLAN.md` の方針に反する実装
- 計画外の仕様追加や勝手な複雑化
- menubar 優先方針に反した UI 先行実装
- Do / Don't に反する進行

### 12.2 処遇フロー

1. 該当フェーズの実装を停止する
2. 何に違反したかを記録する
3. 軽微なら計画追記、重大なら計画再作成とする
4. 緊急例外が必要な場合は 4.5 に従う

---

## 13. AI 向け段階的指示テンプレ

各計画書の末尾に、以下を必ず記載する。

1. SSOT 整合チェック結果
2. Issue → Phase 対応表
3. フェーズごとの Gate 条件
4. 逸脱時の処遇方針
5. 実装順序の最小単位

---

## 14. 計画書レビューサイクル

### Phase 1: AI 作成

- 本ガイドラインの 8 〜 13 を満たす計画書を作成する
- 提出前セルフチェックを計画書に記録する

### Phase 2: 独立レビュー

- 作成セッションとは別コンテキストでレビューする
- 判定は `BLOCKER / ADVISORY / 通過` の 3 段階とする
- BLOCKER が 1 件でもあれば実装開始不可

### Phase 3: 修正

- BLOCKER をすべて解消する
- 変更履歴に修正理由を記録する
- ADVISORY は採否を明記する

### Phase 4: クローズ確認

- BLOCKER 0
- セルフチェック通過
- 必要なら人間確認後に実装承認
